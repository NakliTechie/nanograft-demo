02Wait — is this new?

Every ingredient here is well-trodden. Running Gemma on WebGPU in a browser is common. Training a LoRA is common. Hot-swapping adapters on one base is common — on servers (vLLM, llama.cpp, and friends do it every day).

The specific combination is what seems to be unclaimed ground. I went looking across GitHub, Hugging Face Spaces, Google's own docs, and the wider web for a public demo that runtime-swaps two custom, developer-trained LoRAs on a single in-browser Gemma base — and couldn't find one. The API is documented; the swap-in-a-tab isn't shown.

Every “hot-swap adapters at runtime” demo I could find is server-side. Doing the swap on-device, in a tab appears to be the new part.

Honest caveat: this is absence-of-evidence. A niche, unpublicized demo could exist. But the frontier layer — custom-adapter runtime swap in the browser — has no visible public precedent, while everything underneath it clearly does.

03How it's wired

The trick to shipping a 2.4 GB model demo cheaply: the host serves almost nothing. Cloudflare hands over a few kilobytes of HTML/JS; the heavy bytes come from a CDN and Hugging Face; your GPU does all the actual compute.

Under the hood it's MediaPipe LLM Inference (Web): Gemma-2-2B converted to its int8 GPU format, loaded once, with loraRanks declared up front. Each adapter is a standard PEFT LoRA (attention layers only — q/k/v/o_proj, rank 16) converted to MediaPipe's LoRA format, loaded with loadLoraModel() and selected per generation. Same-origin isn't required; the model streams cross-origin from Hugging Face with a progress bar, and Cloudflare's egress bill stays at essentially zero because it never touches the big files.

04The build log

Getting from “PEFT adapter on disk” to “legible pirate voice on a quantized model in a browser tab” meant walking through a series of quiet traps. In order:

$ nanograft — trap → fix, in the order they bit
  1. trap
    jax-metal breaks the converter. Apple's Metal backend aborts on the converter's StableHLO bytecode. fix uninstall it; run the converter with JAX_PLATFORMS=cpu.
  2. trap
    is_quantized silently turns on SRQ. Static-range quant then demands activation-calibration tensors that don't exist → hard C-level abort. fix pass apply_srq=False.
  3. trap
    The converter ignores lora_alpha. It's accepted and never read, so the browser applies the raw low-rank delta at half the intended strength. fix fold the alpha/r scale into the weights before converting.
  4. trap
    MediaPipe applies no chat template. Feed it a raw prompt and a sharp adapter collapses to an instant end-of-turn. fix wrap the prompt in Gemma's turn format yourself.
  5. trap
    Swap = load-then-use. Pre-loading several adapters and picking by handle silently aliases them all to the last one loaded. fix load the adapter right before you generate with it (it's only ~74 ms).
  6. trap
    Quantization eats the style. int8 washes out sparse lexical quirks (needs a scale nudge); 4-bit degrades the adapter entirely — the base stays coherent but base+LoRA turns to noise. fix ship int8, keep attention layers int8 even in mixed modes.

None of these are in the docs in one place. That's most of what this write-up is for.

05Not vLLM — S-LoRA for the edge

It's tempting to call this “vLLM in the browser.” Half-right. vLLM is two things: a multi-LoRA server (one base, many adapters, different requests using different adapters in the same batch) and a throughput engine (continuous batching, paged attention) that keeps a GPU saturated across many concurrent users.

A browser tab has one user. So the throughput half — the reason vLLM exists — mostly evaporates on-device. What carries over is the other half: the adapter multiplexing. This is S-LoRA's idea (one base, thousands of swappable adapters) brought to the edge — minus the batching.

The axis flips: users → tasks.

On a server you multiplex adapters across people. In a browser you multiplex them across tasks for one person — a per-site profile here, a form schema there, a “my writing voice” everywhere. The scarce resource isn't throughput; it's the one-time base download and VRAM. Pay that once, and every new behavior costs 16 MB and a few dozen milliseconds.

06What it's good for

The superpowers are: private (nothing leaves the tab), $0 per token (your GPU), offline after first load, and one base → many tiny swappable behaviors. That points at a specific shortlist.

the flagship

On-device form-filling

A per-site or per-schema adapter reads a form and fills it — PII never transits a server, works offline, no per-call cost.

per-site

Site-aware assistants

Each site resolves a small profile adapter — its schema, actions, tone — swapped in as you navigate.

compliance

Private text tools

Rewrite, summarize, or redact sensitive text (health, legal, HR) with the guarantee that it stays on the device.

DLP

Client-side extraction

Classify or strip PII from a page before anything is ever sent anywhere.

personal

Your own voice

A “my writing style” adapter for suggestions everywhere, without a training server ever seeing your text.

economics

Edge-scale inference

Push inference to each user's device instead of paying API costs for millions — base once, skills at 16 MB.
